SkyScouter — Jetson Live Camera Pipeline

How a live camera feed from the SIYI A8 Mini becomes a locked, tracked target — and how that same target then steers two actuators in real time: the gimbal camera (UDP) and the aircraft itself via the ArduPilot flight controller (MAVLink). Built to run on a Jetson at the edge, on the drone, no cloud involved.

YOLOv26n TensorRT FP32 · imgsz 1024 Kalman + Inter-frame LK SIYI A8 Mini Gimbal Camera 9-State Lock FSM Pixhawk 6X · ArduPilot · MAVLink Jetson Orin Nano Super 1920 × 1080 · 30 fps

Single-Target Tracker — Kalman + LK Optical Flow
skyscouter/tracking/single_target_kalman_lk.py · SingleTargetKalmanLKTracker

Four-Layer Stack

  • Inter-frame LK thread — tracking/inter_frame_lk.py. Runs LK on every decoded camera frame (~30 fps, 5–10 px steps) in a background thread, not just on YOLO frames. This is the "never loses it" layer — adapted from the engineer's click-to-track code, but driven autonomously by the detector instead of a mouse click
  • Kalman filter — 8D state: cx, cy, w, h, vx, vy, vw, vh. Constant-velocity model
  • Per-YOLO LK — back-checked Lucas-Kanade bridges the gap between detector hits (5–7 fps, 30–50 px steps)
  • Appearance model — HSV-HS histogram 24×16 bins, EMA update, Bhattacharyya similarity
  • Association score: IoU + center dist + area + appearance

Key Params (current config — tuned for tiny, fast targets)

  • Reacquisition radius: 450 px (was 320)
  • Max prediction-only frames: 12 (was 5; ~2 s coast)
  • Track buffer: 60 frames (was 36)
  • LK min points: 4 (was 6; tiny targets)
  • Min confirmed hits: 3
  • Output: List[Track] → pick_primary_track()
How the inter-frame LK is wired: the RTSP source calls register_frame_observer(tracker.feed_inter_frame) so every decoded frame is fed (non-blocking, drops oldest when its queue is full) to the LK thread; the tracker prefers that thread's latest position over the per-YOLO LK, and re-syncs both whenever a real YOLO detection lands. Measured wins: ~3× time in LOCKED, +52% detection rate, 2.8× track lifetime.
primary Track { track_id, detection, hits, time_since_update, center_history[64] }

Lock State Machine
skyscouter/lock/state_machine.py · one independent instance per detected target

Inputs (per frame)

  • is_track_alive, is_track_confirmed
  • class_label — normalized: drone / bird / unknown
  • class_confidence, lock_quality [0–1]
  • bbox_frame_fraction — target size in frame
  • bbox_center_norm_distance — how centered
  • Any fault_flag → immediate ABORTED

Output: LockDecision

  • lock_state — one of 9 states
  • guidance_valid — TRUE only in LOCKED / STRIKE_READY
  • lock_quality, fault_flags, reason
  • TRACKING→LOCKED: 5 consecutive frames passing gates
  • LOCKED→STRIKE_READY: centered + stable ≥ 0.5 s
LockDecision { lock_state, guidance_valid=TRUE only in LOCKED/STRIKE_READY }

6
Guidance Computer
guidance/bearing.py + filtering.py + controller.py
Measures how far the drone is from the frame center, converts that pixel gap into a real angle using the camera's field of view, smooths out jitter with a filter, and predicts where the drone will be 200 ms ahead so the gimbal leads it rather than chasing it.

Outputs a rotation speed for the gimbal. Active from TRACKING onwards.
GuidanceHint { bearing_error_deg, yaw_rate_cmd_deg_s, pixel_error_px, filtered_bearing_error_deg }
7
Gimbal Follow Controller → SIYI A8 Mini
skyscouter/gimbal/follow_controller.py + gimbal/siyi_client.py

Rate Limiting & Safety

  • Max command rate: 10 Hz
  • Deadband X: 60 px, Y: 45 px
  • Active states: TRACKING, LOCKED, STRIKE_READY
  • Stop-after-lost: 0.25 s
  • dry_run=true logs but sends no UDP

Command → SIYI :37260 (UDP)

  • Speaks SIYI SDK v2 — yaw and pitch speed in one packet
  • Gain: yaw 15, pitch 12 (tunable in config)
  • Clamped: yaw ±18, pitch ±15 max speed
  • Every command logged to gimbal_follow_commands.jsonl
The same GuidanceHint that drives the gimbal (stage 7) is also handed to the flight controller (stage 8). One vision pipeline, two independent actuators — neither knows the other exists.
GuidanceHint { bearing_error_deg, filtered_bearing_error_deg } + LockDecision.lock_state
8
MAVLink Flight Link → Pixhawk 6X (ArduPilot)
skyscouter/flight/mavlink_link.py · MavlinkFlightLink  |  built by flight/factory.py  |  Phase 1: yaw-only + alt-hold

What it consumes

  • consume(hint) — gates on guidance_valid + lock_state ∈ {TRACKING, LOCKED, STRIKE_READY}
  • Reads filtered_bearing_error_deg (falls back to bearing_error_deg): + = target is right of centre
  • Applies yaw deadband (1°), then clamps to ±max_heading_deg (15°) per command
  • Stale-guidance timeout 0.5 s → commands zero yaw (hold heading)

What it sends (30 Hz sender thread)

  • Auto-sequence: SET_MODE → GUIDED → force-ARM → NAV_TAKEOFF → settle → stream
  • Per tick: SET_POSITION_TARGET_LOCAL_NED (alt-hold) + CONDITION_YAW (relative)
  • ArduPilot magic numbers: force-arm param2 = 2989.0, alt-hold-only type_mask = 1531
  • dry_run=true computes & logs but sends nothing; live needs --flight-control-live
  • Every command logged to flight_commands.jsonl
↑  CLOSED LOOP × 2 — the gimbal (UDP) re-aims the camera and the aircraft (MAVLink yaw) re-aims the airframe; both change the next RTSP frame, so the pipeline continuously minimises the pixel error between target and frame centre.  ↑
Parallel outputs written every frame: raw_camera.mp4 (written before inference), annotated.mp4 (bboxes + lock-state overlay), operator view (HDMI or MJPEG-over-WiFi), diagnostics.csv, plus gimbal_follow_commands.jsonl and flight_commands.jsonl when those actuators are enabled.

Module Map

Where everything lives, grouped by what it does. Every module can be tested on its own — pipeline.py is the one place they all come together.

📥 Camera Input
skyscouter/io/rtsp_stream_source.pyRuns a background thread that continuously pulls the latest frame from the SIYI A8 Mini over RTSP — so the pipeline always gets a fresh image, never a stale buffered one. Reconnects on its own if the stream drops.
skyscouter/io/frame_source.pyMakes all camera sources look identical to the rest of the pipeline. One value in the config file (source.type) swaps the camera — nothing else needs to change.
skyscouter/io/live_camera_source.pyThe original USB webcam reader from before the SIYI was introduced. Still here but not used in any active config.
🔍 Detection
skyscouter/perception/yolo_ultralytics.pyRuns YOLOv26n on every frame and returns a list of drone candidates with bounding boxes and confidence scores. Works the same way with a .pt file during development or a TensorRT .engine on the Jetson — no code change needed.
skyscouter/perception/factory.pyReads the detector.backend key from config and hands back the right detector. Change one line in YAML to swap models — the pipeline never knows anything changed.
🎯 Tracking
skyscouter/tracking/single_target_kalman_lk.pyKeeps following the drone even when the detector misses it for a few frames. Uses four layers: a background inter-frame LK thread, motion prediction, per-detection optical flow, and color appearance matching to confirm it's still the same object if multiple detections appear nearby.
skyscouter/tracking/inter_frame_lk.py✦ The "never loses it" layer. A background thread runs Lucas-Kanade optical flow on every decoded camera frame (full 30 fps) instead of only on the slower YOLO frames, with a forward-backward consistency check and auto feature-refresh. This is the autonomous adaptation of the engineer's click-to-track technique — same flow idea, but seeded by the detector, no mouse click.
skyscouter/tracking/factory.pySame pattern as the detector factory — tracker backend is a config value, not a code change.
🔒 Lock Decision
skyscouter/lock/state_machine.pyDecides when the system is genuinely confident enough to call it a lock. Pure deterministic logic — no ML, no probability. Fully covered by unit tests. Each detected target gets its own instance, so multiple tracks in the frame never interfere with each other.
🧭 Guidance
skyscouter/guidance/bearing.pyConverts the pixel gap between the drone and the frame center into a real bearing angle, smooths out jitter, then adds a short lead prediction — so the gimbal steers slightly ahead of the drone rather than always trailing behind it.
🎮 Gimbal Control
skyscouter/gimbal/follow_controller.pySits between the guidance output and the actual hardware. Caps commands at 10 Hz so the gimbal isn't overwhelmed, ignores tiny errors below a deadband threshold, and stops the gimbal quickly once the track is gone.
skyscouter/gimbal/siyi_client.pyThe only file in the whole codebase that physically talks to the gimbal. Sends UDP speed commands to the SIYI A8 Mini to rotate the camera. If you want to understand the wire protocol, this is the place to look.
🛩️ Flight Control · NEW
skyscouter/flight/mavlink_link.py✦ The only file that talks to the autopilot — the FC counterpart of siyi_client.py. Takes the per-frame GuidanceHint, turns the bearing error into a relative-yaw command, and runs the SET_MODE → ARM → NAV_TAKEOFF → alt-hold + CONDITION_YAW sequence over MAVLink on a 30 Hz background thread. This is where everything from the old hand_track.py MavlinkYawLink now lives. Start here.
skyscouter/flight/factory.pySame factory pattern as the gimbal/detector — reads the flight_control config block, and returns a started MavlinkFlightLink (or None when disabled). The pipeline never imports pymavlink directly; it's lazy-loaded here only when the link is enabled.
📤 Output & Display
skyscouter/output/annotator.pyDraws the detection box, lock-state color (green = locked, orange = tracking, red = lost), the guidance direction arrow, and the frame latency overlay before writing each frame to the output video.
skyscouter/output/operator_view.pyShows the annotated live feed in real time — either full-screen on a connected HDMI display, or as an HTTP stream you can open from any browser on the local network.
🏗️ Core
skyscouter/schemas.pyAll the data structures that modules pass to each other — what a target state looks like, what lock states exist, what counts as a fault. If two modules need to communicate, the contract is defined here. Change it carefully.
skyscouter/pipeline.pyThe only file that touches all the others. Runs the main frame loop, wires every module together in the right order, and keeps one independent lock-state tracker running per detected target. Start reading here if you're new.

Flight Controller Onboarding — hand_track.py → SkyScouter

For the FC / drone engineer: what moved where from drone-control-jetson-main/, and how you keep building on the new codebase

The one mental shift
Your old hand_track.py was a single ~2,300-line script that did everything in one file: open the camera, run hand detection, smooth the target, run a PID, and talk MAVLink to the autopilot. SkyScouter does the exact same job but splits it into small, independently testable modules wired together by one orchestrator (skyscouter/pipeline.py).

Your MAVLink logic was not rewritten — it was relocated, almost line-for-line, into skyscouter/flight/mavlink_link.py. The magic numbers you already trust (force-arm 2989.0, alt-hold type_mask 1531, relative CONDITION_YAW) are preserved verbatim. What changed is where the target comes from: instead of a hand-detection box you clicked toward, the autonomous vision pipeline produces a GuidanceHint with a bearing_error_deg, and the flight link consumes that.

The MAVLink command flow (unchanged from your code — just relocated)
Exactly the 7 message types your hand_track.py sent — nothing more. The sender thread walks this sequence automatically once consume() starts receiving valid hints:

Bring-up sequence (once)

  • DO_SET_MODE → GUIDED  (sent before any gate — the fix that made option 6 work)
  • COMPONENT_ARM_DISARM force-arm, param2 = 2989.0
  • NAV_TAKEOFF → guided_alt_m (default 2 m; --fc-alt-m to raise)
  • settle: hold yaw=0 for post_takeoff_settle_s

Per-tick stream (30 Hz)

  • SET_POSITION_TARGET_LOCAL_NED, type_mask = 1531 (altitude hold only)
  • CONDITION_YAW, param4 = 1.0 (relative); heading from bearing error, clamped ±15°
  • HEARTBEAT + REQUEST_DATA_STREAM kept alive
  • Stale hint > 0.5 s, or lock lost > stop_after_lost_s → yaw 0 (hold)
No LAND / RTL / disarm in code — same as your hand_track.py. Those stay on the pilot's RC switch as the hardware failsafe. Phase-1 failsafe in software is "zero yaw + hold position" (your Q12 answer).

Camera Intrinsic Calibration — Checkerboard → Bearing Accuracy
Replacing the current simplified FOV-based bearing model with a proper intrinsic matrix from a checkerboard calibration session. Directly improves gimbal servo accuracy and the accuracy of bearing/elevation angles reported in GuidanceHint.
skyscouter/guidance/bearing.py — main change skyscouter/calibration/ — new module (suggested) schemas.py — calibration_id already in TargetState

What changes in bearing.py

  • Current model: bearing_err = pixel_err / frame_width × horizontal_fov_deg — approximation, assumes linear projection
  • Calibrated model: bearing_err = arctan((px − cx) / fx) using intrinsic matrix K = [fx, 0, cx; 0, fy, cy; 0, 0, 1] from the checkerboard session
  • Undistortion step: cv2.undistortPoints() with distortion coefficients before computing angle — especially relevant at SIYI A8 Mini's wide FOV edges
  • Load K and dist coeffs from a calibration file at startup — a YAML or NumPy .npz next to the detector weights works well

What's already in place

  • TargetState.calibration_id field already exists — stamp the calibration file's hash so every output log knows which K was active
  • BridgeProposal.calibration_reviewed bool already defined — flip this once a human has signed off on the calibration quality
  • FaultFlag.CALIBRATION_DRIFT is in schemas.py — raise it if a runtime check detects the calibration may be stale (e.g. reprojection error metric)
  • Config key horizontal_fov_deg: 81.0 can stay as a fallback when no calibration file is present — keeps the sim path working
  • OpenCV's calibrateCamera() output drops straight in — no special format needed
